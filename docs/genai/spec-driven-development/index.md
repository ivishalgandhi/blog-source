# Spec-Driven Development

## The Core Idea

Spec-Driven Development (SDD) inverts the traditional relationship between specifications and code.

**Traditionally:** Code is the source of truth. Specs are written to guide development and then discarded.

**With SDD:** The spec is the source of truth. Code is generated *from* the spec and serves it.

> "In this new world, maintaining software means evolving specifications. The lingua franca of development moves to a higher level, and code is the last-mile approach." — GitHub spec-kit

## Why It Matters Now

Three trends make SDD practical today:

1. **AI reached a capability threshold** — LLMs can reliably translate natural-language specifications into working, tested code. This wasn't true 3 years ago.
2. **Software complexity is exponential** — Modern systems integrate dozens of services. Keeping them aligned with original intent through manual processes fails.
3. **Pace of change is relentless** — Pivots are no longer exceptional; they're expected. SDD makes requirement changes a systematic regeneration, not a manual rewrite.

## What a Spec Is (and Isn't)

A spec is a **structured, behavior-oriented artifact** written in natural language that:

- Expresses functionality as user stories ("As a … I want … So that …")
- Defines acceptance criteria in GIVEN / WHEN / THEN format
- Captures edge cases and technical constraints
- Serves as guidance to AI coding agents

It is **not** a detailed prompt, a JIRA ticket, or a traditional PRD. It is a living contract between humans and AI.

## The Three Levels

| Level | Description | Commitment |
|-------|-------------|------------|
| **Spec-First** | Write a spec before each task; may be discarded after | Low — great starting point |
| **Spec-Anchored** | Spec lives alongside code forever; evolves with the feature | Medium — team alignment needed |
| **Spec-as-Source** | Spec is the primary artifact; code is generated output | High — requires strong tooling |

Most teams should start with **Spec-First** and graduate to **Spec-Anchored** over time.

## The Workflow

```
💡 Idea
  │
  ▼
📋 spec.md      ← what to build and why (human-reviewed)
  │
  ▼
🏗️ plan.md      ← how to build it (human-reviewed)
  │
  ▼
✅ tasks.md     ← ordered, dependency-aware task list
  │
  ▼
💻 Code         ← generated by AI, checked by humans
  │
  ▼
🔄 Production feedback → spec evolves
```

Each stage produces a markdown artifact that is **reviewed by a human** before the next stage begins. You are the pilot; AI is the co-pilot doing mechanical translation.
