# SDD meets Windsurf

If your team uses Windsurf, the SDD concepts map directly onto tools you already know — **Rules, Workflows, and Skills**. This page shows the exact correspondence so you can practice SDD without learning a completely new vocabulary.

## The Mapping

| Windsurf | Spec-Kit | Role |
|----------|----------|------|
| **Rules** (`.windsurfrules`) | `constitution.md` | Immutable project principles the AI always follows |
| **Plan mode** (`.windsurf/plans/`) | `spec.md` + `plan.md` | Requirements → architecture, stored as versioned artifacts |
| **Workflows** (`.windsurf/workflows/`) | `tasks.md` | Executable, ordered step sequences |
| **Skills** | Spec-kit templates | Reusable structured prompt patterns |
| **`.windsurf/`** directory | **`.specify/`** directory | Project-level AI configuration root |

## Rules = Constitution

Both encode team-wide, immutable principles that every AI interaction must respect.

**`.windsurfrules` (Windsurf):**
```markdown
## Code Style
- No abstractions for single-use code
- Match existing style — don't "improve" adjacent code
- Minimal changes only: every changed line traces to the request

## Testing
- Write tests before implementation
- Tests must fail before implementation begins
```

**`constitution.md` (spec-kit):**
```markdown
## Article III: Simplicity Over Cleverness
- No abstractions for single-use code
- Maximum 3 top-level modules
- Junior engineer must understand in 5 minutes

## Article II: Test-First Development
- Tests exist and fail (red) before implementation
- Tests document intent; code documents implementation
```

**Key difference:** The constitution is structured with Articles and an Amendment Log. This makes the evolution of principles visible and auditable. Consider adopting this structure for your `.windsurfrules`.

## Workflows = Tasks

Both are ordered, executable step sequences the AI follows.

**`.windsurf/workflows/run-checks.md`:**
```markdown
---
description: Run all health checks and generate a report
---
1. Ensure Docker Compose is running
2. Run pytest to verify all checks pass
3. Start FastAPI server
4. Run `scanner report` CLI command
5. Show output
```

**`tasks.md` (spec-kit):**
```markdown
## [T09] Wire FastAPI routes
**Depends on:** T04, T06, T08
- [ ] GET /health/db calls check_connectivity()
- [ ] GET /health/latency calls check_latency()
- [ ] GET /health/scan calls run_full_scan()
**Status:** 🔄 READY
```

**Key difference:** `tasks.md` tracks *implementation* progress with dependency chains and constitution compliance gates. Workflows are for *operational* step sequences (deploy, test, run). Both are valuable — they serve different purposes.

## Skills = Spec Templates

Both provide reusable structured patterns that prevent starting from scratch.

**Windsurf Skill:** A YAML/markdown blueprint for a common coding task ("create a REST endpoint") that encodes your team's conventions and injects them into every generation.

**Spec-kit Template:** A markdown template for `spec.md`, `plan.md`, or `tasks.md` that ensures every feature gets the same structure, same gates, same checklist sections.

Both reduce cognitive overhead and enforce consistency across a team.

## Plan Files = spec.md + plan.md

Windsurf's plan mode (the `.windsurf/plans/` directory) is the closest Windsurf equivalent to spec-kit's specify + plan phases.

**`.windsurf/plans/my-feature-abc123.md`** (Windsurf plan):
- Generated by Cascade plan mode
- Contains proposed implementation steps
- Reviewed before execution

**`.specify/001-health-scanner/spec.md` + `plan.md`** (spec-kit):
- `spec.md`: functional requirements, user stories, acceptance criteria
- `plan.md`: technical architecture, rationale, file structure
- Both reviewed before tasks are generated

The spec-kit approach separates *what* (spec) from *how* (plan), which is a meaningful discipline.

## Real Example: This Repository

The `.specify/auth-for-docs/` directory in this repo is a working example of SDD you've already done:

```
.specify/
└── auth-for-docs/
    ├── spec.md    ← 5 user stories + 5 edge cases (Google OAuth for Docusaurus)
    ├── plan.md    ← AuthContext, DocRoot swizzle, LoginPage architecture
    └── tasks.md   ← T01–T13 with dependencies and status tracking
```

Open `spec.md` and you'll see GIVEN/WHEN/THEN acceptance criteria. Open `tasks.md` and you'll see test-first task ordering. You practiced SDD without realising it.

## Recommended: Adopt the Constitution Pattern

Regardless of whether you use spec-kit, consider formalising your `.windsurfrules` into a constitution-style document:

```markdown
# Team AI Constitution

## Amendment Log
| Date | Article | Change | Author |
|------|---------|--------|--------|
| 2026-04-20 | I | Added spec-first requirement | Vishal |

## Article I: Spec-First
No implementation without a spec. Workflow: spec → plan → tasks → code.

## Article II: Test-First
Tests written and failing before implementation begins.

## Article III: Simplicity
No abstractions for single-use code. No future-proofing.
```

This gives your rules file the **structure, auditability, and intentionality** of a team agreement — not just a list of AI instructions.
